## 0. 30 秒版（先给结论）

- **DiffusionDrive** 原本是一个**车端自动驾驶的"学习型轨迹规划器"**：看一眼传感器，直接吐出未来几秒该怎么走的路线。它是 CVPR 2025 Highlight，卖点是"用扩散模型生成轨迹但跑得很快"。
- 团队把它**几乎原封不动地**搬上了 SVT 四轮巡检机器人。最关键的一招是**偷换概念**：把自动驾驶里"左转/直行/右转"的指令槽位，改成机器人的**四个巡检任务模式**（巡航 / 转向 / 巡检 / 退出）。
- 模型主体没怎么动，真正新写的是**数据管线**（把机器人录的东西变成训练样本）和**实时部署节点**（在机器人上每秒跑模型、把结果变成底盘速度指令）。
- 软肋有两个：训练用的"标准答案"是**遥控指令积分出来的伪轨迹**（不是真定位），而且**没有一个能闭环测试的环境**。—— 这正好是我们这个仿真器项目要补的两块。

## 1. 先补几个概念（不熟的话看这节）

### 1.1 什么是"学习型规划器"

传统机器人导航是**手写规则**：`if 前方有障碍: 转弯`。
学习型规划器换了个思路：**给神经网络看大量"人类是怎么开的"例子，让它自己学会"看到这种场景该怎么走"**。这叫**模仿学习（imitation learning）**。
DiffusionDrive 就是这种——输入传感器，输出"未来 4 秒的行驶路线"，全靠从人类演示里学。

### 1.2 什么是 BEV（鸟瞰图）

模型不直接"看"原始 3D 点云，而是先把周围障碍物**拍扁成一张俯视的 2D 地图**（Bird's-Eye-View）：就像从机器人正上方往下看，哪里有东西、哪里是空地。
具体做法：把深度/点云里的点按 (x, y) 位置丢进一个网格，**每个格子统计落了多少个点**，点多的格子=有障碍。这张图就是模型"思考"用的地图。

### 1.3 什么是"轨迹"

模型的输出。这里的轨迹 = **未来 8 个时间点的位置**，每点间隔 0.5 秒，一共覆盖 4 秒。
每个点是 `(x, y, θ)` = 相对机器人当前位置的 前进距离、左右偏移、朝向角。
8 个点连起来就是一条"接下来该这么走"的路线。

### 1.4 什么是"锚点"（anchor）和"扩散"（diffusion）

这是 DiffusionDrive 的核心技巧，分两层：

- **锚点**：与其让模型凭空画一条路线（难、不稳），不如先准备 **20 条"典型路线模板"**（对历史数据做聚类得到，比如"直行""左缓弯""急停"…）。模型从这 20 条模板出发去**微调**，学起来更稳、更快。
- **扩散（截断扩散）**：借用了 AI 画图里"扩散模型"的思想——**先给模板加噪声，再让模型一步步去噪、还原成干净的路线**。原版扩散要去噪几十步很慢；DiffusionDrive 的创新是**只从锚点附近起步、只去噪 2 步**（叫"截断扩散"），所以能做到 **45 FPS 实时**。这是它论文的主要贡献。

> 一句话：**20 条模板 + 2 步去噪 = 又快又多样的轨迹生成器**。

### 1.5 什么是"条件模仿"（这是搬上机器人的关键）

同一个网络，**喂给它一个"指令标签"，它就表现出对应的行为**。
- 自动驾驶里：标签 = `左转 / 直行 / 右转`（导航系统在路口给的高层指令）。同一个网络，给"左转"标签就往左规划，给"直行"就往前。
- 这个机制叫 conditional imitation：**不用为每种行为训练一个网络，一个网络靠标签切换**。
- **后面会看到：机器人版就是把这个"标签槽"从导航指令换成了巡检任务模式。**

## 2. 起点 vs 目标：车和机器人差在哪

| | 原版 DiffusionDrive（车端） | SVT 巡检机器人 |
|---|---|---|
| 跑在什么数据上 | NavSim（nuScenes/nuPlan 自动驾驶基准） | 自己用机器人录的数据 |
| 传感器 | 多路相机 + 真 LiDAR | 1 个 ZED 相机（含深度）+ 2 个 D405 |
| "标准答案"轨迹 | 带高精定位的行车日志（真值） | **没有定位**，只有遥控指令记录 |
| 高层指令 | 左/直/右导航指令 | 想要：巡航/转向/巡检/退出 |
| 速度尺度 | 几十 km/h | 0.1 m/s（很慢，室内巡检） |
| 底盘 | 阿克曼前轮转向 | **四轮独立转向（能横移）** |
| 怎么用 | 离线跑分（benchmark） | **要在机器人上实时跑** |

差距不小，但团队的策略是"**能不改模型就不改，靠适配输入输出和数据管线把鸿沟填上**"。

## 3. 七个适配动作（移植的全部内容）

下面每一条都是"原来是什么 → 改成什么 → 为什么"。

### ① 偷换条件槽位 —— 最精妙的一招（零模型改动）

模型有个输入叫 `status_feature`，是个 8 维向量：

```
status = [ 指令(4维) , 速度vx,vy(2维) , 加速度ax,ay(2维) ]
                ↑
         模型里就是一层 nn.Linear(4+2+2 → 256)
```

- **车端**：前 4 维 = 导航指令 one-hot（左/直/右/未知）。
- **机器人**：前 4 维**原样不动地改成了四个任务模式 one-hot**：

  | 模式 | 编码 | 含义 |
  |---|---|---|
  | cruise 巡航 | `[1,0,0,0]` | 走廊里直行 |
  | turn 转向 | `[0,1,0,0]` | 拐弯、横向对齐 |
  | inspect 巡检 | `[0,0,1,0]` | 慢速靠近、面对货架、原地检查 |
  | exit 退出 | `[0,0,0,1]` | 离开巡检位、恢复巡航 |

**模型结构一行都没改**（还是那个 `Linear(4+2+2)`），只是把这 4 维的"含义"换了。
→ 这就是用**条件模仿**实现"一个网络、四种巡检行为"。
（代码：`navsim/agents/diffusiondrive/transfuser_features.py:45` 原版用 `driving_command`；
机器人版在 `robot_data_prep/task_modes.py` 把这 4 维填成任务模式。设计说明见 `DiffusionDrive-main/docs/ROBOT_BEHAVIOR_TRAINING.md`。）

### ② 传感器映射 —— 让 ZED 假装成"相机+LiDAR"

模型要两路输入：`camera_feature`（RGB 图）和 `lidar_feature`（BEV 地图）。机器人没有 LiDAR，怎么办：

```
ZED 彩色图 ──缩放──────────────────────────→ camera_feature  (3, 256, 1024)
ZED 深度图 ──反投影成3D点──→ 点云 ──拍扁成俯视直方图──→ lidar_feature   (1, 256, 256)
```

- 用 ZED 的深度**算出每个像素的 3D 坐标**，凑成一团点云，**当作 LiDAR 用**，再做成 BEV 直方图。
- 输出形状（256×1024 的图、256×256 的 BEV）**和原版一模一样**，所以骨干网络 ResNet-34 不用改。
- （代码：BEV 生成在 `robot_data_prep/bev.py` 的 `points_to_bev_tensor`；深度转点云在部署节点 `run_diffusiondrive_policy.py` 的 `_depth_to_xyz_m`。）

### ③ 监督信号 —— 没有定位，就用遥控指令"攒"出轨迹

训练模仿学习需要"标准答案轨迹"。车端有定位真值；机器人**没有定位系统**。
他们的办法：把**人遥控机器人时发出的速度指令**（`svtrobot_cmd.jsonl` 里的 `vx, vy, wz`）**按时间积分**，攒出一条"机器人大概走过的路"：

```
每一小段 dt: 
  位置 x += (vx·cosψ − vy·sinψ)·dt
  位置 y += (vx·sinψ + vy·cosψ)·dt
  朝向 ψ += wz·dt
```

然后取"当前时刻之后 0.5s、1.0s … 4.0s"这 8 个点，转换到机器人自身坐标系，就是那条要学的目标轨迹。
（代码：`robot_data_prep/cmd_integrate.py`。）

> ⚠️ **这是最大的妥协**：这条轨迹不是真定位测出来的，而是"指令积分"出来的**伪里程**——本质是在**模仿遥控员当时怎么打的指令**，会累积漂移、也继承遥控员的习惯/抖动。

### ④ 关掉自动驾驶专属的辅助任务

原版除了出轨迹，还顺带学"地图语义分割""检测周围车辆"等辅助任务（帮助训练）。机器人**没有高精地图、也不检测车辆**，所以一个开关关掉：
`use_auxiliary_supervision: false`（配置 `diffusiondrive_robot_agent.yaml`）。只保留轨迹 + 扩散这条主线。

### ⑤ 新增"任务模式预测头"（唯一的模型增量）

车端的导航指令是导航系统给的；机器人希望**不靠人也能自己判断现在该巡航还是该巡检**。
于是加了个小模块 `mode_head`：**从 BEV 特征预测当前该用哪个任务模式**（4 选 1，置信度够高才切换）。
这样机器人可以自主运行；需要时操作员/上层也能强制指定模式。
（这是相对原版**唯一新增的网络模块**，配置里多出 `mode_num_classes: 4` 等几行。代码在 `transfuser_model_v2.py`。）

### ⑥ 全新数据管线 + 重算锚点（绕开 NavSim 的那套加载器）

原版读 nuScenes 的复杂数据格式。机器人**另起炉灶**，新增 4 个文件：
- `robot_data_prep/`：把一次录制（图+点云+指令）**转成训练样本 `.npz`**。
- `run_training_robot.py` + `robot_dataset.py`：直接读这些 `.npz` 训练，不碰 NavSim 的数据系统。
- `robot_training.yaml` / `diffusiondrive_robot_agent.yaml`：机器人专属配置。
- 锚点也可以用 `compute_robot_plan_anchors.py` 对机器人自己的轨迹**重新聚类**得到（也可沿用车端的 `kmeans_navsim_traj_20.npy`）。

### ⑦ 全新实时部署节点（NavSim 只能离线跑分）

原版只能离线评测。机器人要**真在硬件上每秒跑**。于是写了 `run_diffusiondrive_policy.py`（2600+ 行的 ROS2 节点），干这些事：

```
循环(约每秒一次):
  读 ZED 图+深度 → 做成 RGB + BEV
  喂模型 → 得到 20 条候选轨迹 + 选中的任务模式 → 取最优 1 条(8个点)
  把轨迹的点差分换算成速度 (vx, vy, wz)
  做平滑/限速/深度安全急停(太近就停+倒车)
  发 Twist 指令到 /svtrobot_cmd 话题 → 底盘执行
```

速度被压得很低（`max-vx 0.1 m/s`），适配室内慢速巡检。
（注意：把 `(vx,vy,wz)` 变成"四个轮子各转多少、各打多少角"的**四轮转向逆运动学在底盘固件里**，不在这个 Python 节点——所以上层只管发 `(vx,vy,wz)`。）

## 4. 数据是怎么变成训练样本的（一图流）

```
一次录制 session/
  images/zed/*.jpg        彩色图(每张的文件名=微秒时间戳)
  pointcloud/zed/*.npz    点云
  svtrobot_cmd.jsonl      遥控速度指令(vx,vy,wz)
  task_modes.jsonl        (可选)人工标的"哪段是巡航/巡检…"
        │
        │  robot_data_prep 预处理
        ▼
  每张 ZED 图 = 一个训练样本，打包成一个 .npz：
  ┌─────────────────────────────────────────────────────────┐
  │ image_u8     (256,1024,3) uint8   该时刻的 RGB 图          │
  │ lidar_feature(1,256,256) float32  该时刻点云做成的 BEV     │
  │ trajectory   (8,3) float32        之后4秒的8个点(指令积分)  │
  │ status       (8,)  float32        [任务模式4维, vx,vy, ax,ay]│
  │ task_mode    字符串               "cruise"/"turn"/...       │
  └─────────────────────────────────────────────────────────┘
        │
        ▼
  manifest.csv（列出所有样本）→ run_training_robot.py 读它训练
```

**任务模式从哪来**：优先看人工标注 `task_modes.jsonl`（标了哪段是什么模式）；
没标的用**运动启发式**自动猜（`task_modes.py`）：
- 几乎不前进但在转 → `inspect`（原地检查）
- 明显横移 → `turn`
- 正常前进、不太转 → `cruise`
- 之前一直在转、突然变成直行 → `exit`

## 5. 训练时模型在学什么（损失函数）

训练时同时优化几项（`transfuser_loss.py`）：
- **轨迹回归**：预测的 8 个点要接近"标准答案"轨迹（L1 距离）。
- **扩散损失**：去噪过程要学对。
- **多模分类**：20 条候选里，要学会给"最接近答案的那条模板"打高分。
- **任务模式分类**：`mode_head` 预测的模式要对（focal loss，4 类）。
- （车端的地图/车辆检测损失：机器人关掉了。）

配置：学习率 6e-4、batch 4、100 轮、混合精度。

## 6. 软肋 + 对我们项目的意义

这套系统已经很完整（能采集、能训练、能上车实时跑），但有两个结构性软肋——**正好是我们仿真器项目要补的位**：

### 软肋 A：训练的"标准答案"是伪里程（模仿遥控）
轨迹标签靠遥控指令积分得到，不是真定位 → 会漂移、会继承遥控员的抖动和习惯，质量天花板有限。
**我们能补**：仿真器里**机器人位置是程序已知的真值**，能产出干净、准确的轨迹标签；还能**脚本化地造各种巡检场景**（各种 cruise/turn/inspect/exit），任务模式标签也天然干净。→ 用我们 sim 产的 `.npz` 训练样本，质量可能比真机伪里程更好。

### 软肋 B：没有闭环测试环境
真机现场测试又慢又危险；NavSim 离线跑分又对不上这个机器人。训好的模型**没地方安全地验证**。
**我们能补**：我们的**双层仿真器**正好做这个——
- 几何层 → 产出和真机同格式的 **BEV (1,256,256)**
- 3DGS / ArtiFixer 皮肤 → 产出 **RGB (3,256,1024)** 当第一视角
- 加载他们训好的权重 → 出轨迹 → 用 `(vx,vy,wz)` 驱动仿真底盘 → **闭环跑起来看撞不撞、到不到位**

### 一句话定位
> 他们做出了"快脑"（DiffusionDrive 巡检策略）的真实落地；**缺的是一个能闭环测试、又能产干净训练数据的环境——那就是我们这个项目**。这和我们既定的"VLM 慢脑 + DiffusionDrive 快脑"双脑架构完全咬合：他们现在用一个小 `mode_head` 从场景猜任务模式，**未来由更强的 VLM 来设这个模式**。

## 附：和我们仿真器对接的"数据契约"（要做闭环时照这个产）

| 项 | 形状 / 格式 | 说明 |
|---|---|---|
| RGB 输入 | `(3, 256, 1024)` | 第一视角；可用 3DGS/ArtiFixer 皮肤渲 |
| BEV 输入 | `(1, 256, 256)` float32 | 点云直方图，范围 ±32m、4 像素/米、z>0.2m、每格点数封顶5归一化 |
| 任务模式 | 4 维 one-hot | cruise/turn/inspect/exit（未来由 VLM 设） |
| 轨迹输出 | `(8, 3)` ego帧 | [x前, y左, θ]，0.5s 间隔共 4s |
| 控制接口 | `Twist(vx, vy, wz)` → `/svtrobot_cmd` | 当 omni 底盘积分即可，逆运动学在底盘固件 |

> 坐标约定：相机系(右,下,前) → ego系(前,左,上)。
