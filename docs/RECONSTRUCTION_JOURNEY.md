## 1. 真实数据实情(和原计划假设完全不同)

原始规划文档假设数据是 `color/000000.png` + `depth/000000.png` 这种逐帧 PNG。**实际完全不是**。6 个 session(105731 / 111450 / 112633 / 113628 / 114830 / 142522),每个 ~15 分钟,采于 2026-06-11。单 session 结构:

| 数据 | 实情 | 能不能用 |
|---|---|---|
| `images/zed/*.jpg` | ZED 左目彩色,时间戳(ns)命名,1280×720,~1880 张 | ✅ 重建用它 |
| `images/zed_right/*.jpg` | ZED 右目(立体) | ✅ |
| `images/d405_1/*.jpg` | RealSense D405(**机械臂相机**) | ❌ 用户明确**排除** |
| `depth/*/*.jpg` | **伪彩有损深度可视化**(3通道JPEG) | ❌ 不是真深度,丢弃 |
| `pointcloud/zed/*.npz` | ZED SDK 稠密有序点云,`xyzrgba` (720,1280,4) float16,**真实米制 XYZ** | ✅✅ 几何主料 |
| `imu.jsonl` / 单独文件 | 高频 accel/gyro/mag(**无融合四元数**) | ✅ 朝向用 |
| `rosbag*/*.db3` | 只有 `/svtrobot_cmd`(Twist)、`/f710/joy`、`/chassis/joint_states`(4轮转向编码器)、`/chassis/diagnostics` | ⚠️ 仅控制/底盘 |

### 🔴 决定全局的三个"缺失"
1. **没有 .svo 原始文件** → ZED SDK 原生 VIO/空间建图(本来最优)**用不了**;全盘 `find` 无 .svo,采集流程也没存。
2. **rosbag 里没有图像/IMU/odom/tf/scan 任何一个 topic**(实测确认:只有 chassis/joy/cmd)→ 任何"重放 bag 跑在线 SLAM"的方案**无数据可喂**。
3. **彻底没有位姿/里程计**。`/svtrobot_cmd` 只是**期望速度指令**(开环,积分必漂);4WS 底盘轮速需标定;IMU 无姿态。**位姿必须自己从视觉重建推**。

> ⚠️ 还有一个坑:早期解出来的 `*.jsonl`(svtrobot_cmd 空、f710_joy 全0、chassis 截断)是 **bag 转换器坏了**,不代表机器人没动。真实数据在 `.db3` 里,要用 sqlite 直接 CDR 解码。基于坏 jsonl 的判断全部作废过一次。

## 2. 走过的弯路(诚实清单)

按时间顺序,每条都说清"为什么试 / 为什么失败 / 学到什么"。

### 弯路 ①:手画/随机生成的假场景
- **做了什么**:最初的 `sim2d` 用 `np.random` 生成墙、目标、路线。
- **为什么废**:与真实数据零关系,一眼假。用户明确否决"造假"。
- **教训**:几何必须锚定真实数据,这是底线。

### 弯路 ②:手写 ICP / 轮速 / cmd 三套里程计,都过不了"漂移关"
- **做了什么**:① 帧间 point-to-plane ICP 累积位姿;② `/chassis` 轮速里程计;③ 积分 `/svtrobot_cmd` Twist(dead reckoning)。
- **为什么废**:全是**纯航位推算,无位置反馈,误差只增不减**。开环积分 cmd 角速度积出 11668°(荒谬);过 U 形掉头时朝向误差被带进返程,返程不重合去程而是斜着偏出。室外单向长穿行**没有回环可闭**,误差无法纠正。
- **教训**:序列累加的里程计在长序列 + 无回环时必漂;需要**全局**一致的解法。

### 弯路 ③:Open3D 位姿图 / RTAB-Map 式回环
- **做了什么**:Open3D 位姿图(里程计边 + 按位置找回环候选)。
- **为什么废**:室外基本单向穿行,**只找到 1 条回环边**,全局优化没东西可纠;漂移又把"按位置找回环"本身搞乱。
- **教训**:回环闭合的前提是真的走回同一地点,这份数据不满足。

### 弯路 ④:朝向直接积分 gyro_z
- **为什么废**:相机/IMU 下倾 ~15°,直接积分 `gyro[2]` 得净转 3°(废)。
- **修正**:重力 ≈ `mean(accel)` 单位向量;`yaw_rate = gyro · ĝ`(投影到重力轴)去偏置后积分 → 真实净转 -108°。**这条最后被采用,复用于所有 session**。

### 弯路 ⑤:distance 传感器当"真实激光射线"
- **为什么废**:111450 的 front 全程恒定 ≈1825mm、left ≈95mm,730 秒室内外都不变 → 像在测机器人自身固定结构,不是环境。
- **教训**:"黄色射线=真实墙距"前提不成立,射线改为在重建出的墙上**做射线投射**。

### 弯路 ⑥:3D 高斯泼溅(3DGS)做几何 / 俯视图
- **做了什么**:对 ZED RGB + COLMAP 位姿训 3DGS(成功出 92 万高斯、照片级,`gs_111450.ply`)。想"取 BEV 密度还原 2D"或"正俯视/环绕渲染"。
- **为什么废**:① **3DGS 优化的是渲染不是几何**,近相机有大量 floater(伪墙正好压在路径上);② **3DGS 只在训练走过的视角附近有效**,头顶/环绕这种没观测过的视角渲染出来**全是黑+噪点**(地面前视数据天然没有头顶信息)。
- **留下的有用东西**:`gs_111450.ply` + **沿真实轨迹的第一视角"穿行回放"** `drivethrough_111450.gif`(照片级清晰,这是 3DGS 唯一好看的用法,可当 demo)。
- **教训**:**深度数据要用几何工具(TSDF),不要用渲染工具(3DGS)做几何**;前视数据给不出俯视图。

### 弯路 ⑦:尺度标定用 cmd 路程 —— 错了 11 倍!
- **为什么废**:用 `/svtrobot_cmd` 积分路程标定出 scale=0.762,导致"场景 25m / 机器人 0.045m/s"——荒谬。
- **修正**:**ZED 双目深度本就是米制真值**。用 ZED 深度对 COLMAP 点深度求比值(15463 样本)→ **1 COLMAP 单位 = 8.276m**。真实场景 ~270m、~0.47m/s,合理。**以后一律用 ZED 深度标定尺度**。

### 弯路 ⑧:2D 矢量化用 HoughLinesP / 密度阈值
- **为什么废**:散点喂 `HoughLinesP` 会强行连成乱线;"密度阈值挑立面"看似成功,其实**留下了路径上的伪影、滤掉了稀疏的真墙**(假象)。
- **修正**:形态学 `膨胀→闭运算→findContours→approxPolyDP` 老实描线;最终方案见第 3 节。

### 弯路 ⑨(本次):开放墙用"局部行进方向"分左右
- **为什么废**:111450 轨迹是**同一走廊来回约 13 趟巡逻**(总弧长 739m,首尾直线仅 56m)。机器人每次掉头,"左/右"就翻转 → 两侧墙不停互换 → 墙锯齿交叉成一团(比上一版更差)。
- **修正**:用**全局走廊主轴**(轨迹 PCA 主向)的垂直偏移分侧(不受掉头影响)→ 分箱取中位 → 平滑 → **两道干净开放 facade**。✅ 当前结果。

### 弯路 ⑩(本次):COLMAP CPU 匹配崩溃 + RTAB-Map 装不上
- **COLMAP 崩溃**:这台预编译 pycolmap **无 CUDA 匹配**,只能 CPU faiss;多线程 OpenBLAS 内存竞争 → SIGSEGV。修:`OPENBLAS_NUM_THREADS=1`(消竞争)+ `OMP_NUM_THREADS=8`(worker 仍并行)+ `cpu_brute_force_matcher`。
- **RTAB-Map**:conda-forge / robostack **都没有** rtabmap 包,无 sudo 又用不了 apt → **这台机器装不了 RTAB-Map**。改用 RoboStack(免 sudo,装到 home)装 ROS2 + `slam_toolbox` / `octomap_server`。

### 弯路 ⑪(本次):用标准 ROS2 占据建图(octomap_server)对比 —— 一坨黑斑
- **做了什么**:装好 ROS2(RoboStack)后,把 COLMAP 每帧位姿 + ZED 米制点云逐帧喂给标准工具 `octomap_server`(tf map→camera + PointCloud2,让它做真正的射线投射),出标准 `.pgm` 占据图。
- **结果**:**糊满走廊的一坨黑斑**(占据 4.8万格 / 自由仅 0.8万格),调"只投建筑高度 1.5~6m"也一样。**比我们自研的双墙提取更差**。见 `compare_octomap_vs_sim2d.png`。
- **为什么**:前视相机把地面/路径/建筑全投成占据,稀疏帧又 carve 不出干净自由空间。**这恰恰证明了"地图干不干净由场景决定、不由工具决定"**——换 RTAB-Map(内部同样是占据投影)对这份数据也会是类似的一坨。
- **价值**:这是一次**有证据的证伪**——"换个标准 SLAM 工具就干净了"在这份数据上不成立。但它也给了启发:octomap 保留真实场景范围是对的,只是把地面也投了 → 见下面的"竖直跨度占据图"(把这条思路做对了)。

### 弯路 ⑫(本次):114830 用 exhaustive 匹配 —— 2 小时才一半且越来越慢
- **为什么慢**:315 张图 exhaustive(全两两配对)在无 CUDA 的 CPU 上,跑 2 小时才到 [3/7] 块,单块还从 167s 涨到 855s。预计总共 5-6 小时。
- **修正**:这是**连续视频**,改 `match_sequential`(overlap=20 + quadratic,只配时间邻居+指数间隔帧,复用已提特征)→ **30 分钟匹配完**,注册 291/315 张、5.7 万点,质量足够(走廊连续重叠 + 往返在掉头处自然相接)。
- **教训**:连续视频用顺序匹配,别用 exhaustive;exhaustive 只在无序图集才必要。

## 3. 最终走通的技术路线

```
ZED 左目 jpg ──► COLMAP 全局 SfM(extract → 顺序匹配 → incremental_mapping,全局BA)
                   │  连续视频用 match_sequential(比 exhaustive 快~10×);全局BA解决漂移/返程连接
                   ▼
              干净的相机位姿(COLMAP 坐标,任意尺度)
                   │  ×尺度(每 session 自算 = ZED深度/COLMAP深度比值;111450=8.28、114830=4.67)
                   │  clean_camera_mask 剔除误注册相机(PCA主轴+垂直残差)
                   ▼
ZED npz 米制深度 ──► TSDF 融合(Open3D ScalableTSDFVolume,voxel0.25,trunc12m)
                   │  几何工具,无 floater(对比 3DGS)
                   ▼
              干净走廊点云 tsdf_<s>.ply + 重力对齐
                   │
        ┌──────────┴──────────────────────────────┐
        ▼                                          ▼
 【分支A:抽象 facade】                      【分支B:标准占据图 ⭐推荐】
 open_walls():全局走廊主轴分两侧            occgrid_vertical():每 0.2m 格算竖直高度跨度
 → 分箱中位 → 平滑 → 两道开放折线           → 跨度>1.2m=竖直立面=占据,<0.4m=平地=可行驶
        │                                          │  (鲁棒于室外地面倾斜,自动滤掉地面涂抹)
        │                                   clean_occgrid():闭运算连断点+去噪点斑
        ▼                                          ▼
 sim2d_<s>_map.png/.gif                     occ_vert_<s>_clean.pgm/.yaml
 + walls_<s>.json                           (标准 ROS 占据图,nav2/Stage 直接加载)
 (游戏感抽象世界,带激光)                   (忠于真实场景范围 + 两侧立面 + 去地面)
```

**为什么是这套**:无 .svo、无 odom、手写里程计漂 → COLMAP 全局 BA 是这些约束下**正确且成功**的位姿来源;深度是米制 → TSDF 是正确的几何工具;室外开阔 → 墙做成**开放**结构而非闭合房间。

**两种 2D 交付形态(都从同一份 TSDF 几何出,可按用途挑)**:
- **分支 A(抽象 facade)**:墙=线段,最像学长参考图的"游戏感世界",适合放算法 agent + 激光演示。114830 这版最干净(单条 ~6m 宽走廊)。
- **分支 B(标准占据图,⭐当前看好)**:`.pgm`/`.yaml` 标准 ROS 格式,**用"竖直高度跨度"判障碍**——墙/建筑竖直(跨度大)、地面平(跨度小),自动分开。比 octomap 原始一坨干净,又比抽象双线更忠于真实场景范围,且 nav2/Stage/Flatland 直接能加载跑导航。**这是"标准工具的忠实 + 去掉地面涂抹"的正解。**

## 4. 关键硬约束与结论(都有证据,会改变方案)

1. **6 个 session 没有一个能拼出"干净多房间室内图"**。抽样近墙点:105731/111450/112633/113628/114830 ≈ 0(开阔/室外),142522 有墙但**近乎静止对着一面墙**(占据图是单向扇形非房间)。
2. **真实墙提不出干净轮廓——是几何本质,不是后处理 bug**。叠图验证(`validate_111450.png`):路径附近最密的"墙"亮脊正好压在轨迹上 = **3DGS/前视近相机伪影**;离轨迹 >1.2m 的**真实两侧建筑太稀疏、连不成线**。
3. **尺度用 ZED 米制深度标定**(8.276 m/单位),不要用 cmd 路程。
4. **朝向用 gyro 投影重力轴**,不要直接积分 gyro_z。
5. **几何用 TSDF,不要用 3DGS**;**位姿用 COLMAP 全局 BA,不要手写里程计**。
6. **地图干不干净由场景决定,不由工具决定**(已用 octomap_server 实测证伪,见弯路⑪)。标准工具开箱即用 → 糊满走廊一坨。
7. **但"标准占据图"这条思路是对的,关键是障碍判据**:不要把所有点都投成占据(地面会糊),要按**每格竖直高度跨度**分(墙竖直/地面平)。这样出的 `.pgm` 既忠于真实场景范围、又干净、还是 nav2/Stage 直接能用的标准格式 —— **当前看好这条作为主交付**。
8. **连续视频用顺序匹配**(match_sequential),别用 exhaustive(CPU 上慢 ~10×)。

## 6. ⭐ 下次数据采集规范(本次最值钱的教训)

**这件事在"采集当下"是 trivial 的,在"事后离线重建"才难十倍。** 下次采集务必:

1. **录 ZED 的 `.svo` 原始文件**(开 positional tracking)→ 直接拿带尺度轨迹 + 空间建图,跳过 COLMAP/TSDF 整条链。
2. **采集时实时跑 SLAM**(`slam_toolbox` / `cartographer`),`/map` topic 直接就是标准 `.pgm` 占据图。
3. **rosbag 录全 topic**:`/odom`、`/tf`、`/zed/.../image_raw`、`/zed/.../camera_info`、`/imu`、`/scan`(不要只录 cmd/joy/chassis)。
4. 若做室内规整地图:让机器人**正对墙、慢速、多走回环**(回环闭合需要真的走回同一地点)。

> 有了 1–3 任何一条,本文 90% 的弯路都不会发生。
