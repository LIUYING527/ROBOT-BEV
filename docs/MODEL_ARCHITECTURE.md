## 0. 一句话定位

**把 DiffusionDrive（车端自动驾驶的扩散轨迹规划器）迁移到「任务驱动型机器人巡检导航」**，
上面加一层 VLM 做场景/任务识别。这是项目的核心 framing 与论文级贡献点。

## 1. 总体架构：双层 / 双脑

```
┌───────────────────────────────────────────────────────────────┐
│  慢脑  VLM (Qwen3-3B 级)            频率 ~0.2 Hz (~5s/次, 6-8张图) │
│        职责: 场景识别 / 任务路由                                  │
│        输出: 离散任务 id  (✅ 已定用离散 id，不用语言 embedding)   │
│        原则: 不做硬标注、不依赖精确坐标、不建米制地图；           │
│              只提供「场景之间的区分度」                           │
└───────────────────────────────┬───────────────────────────────┘
                                 │  task id（缓存，不阻塞下层）
                                 ▼
┌───────────────────────────────────────────────────────────────┐
│  快脑  DiffusionDrive               频率 高频实时 (车端 45FPS@4090) │
│        输入: BEV/感知特征 + ego 状态 + 【VLM task id】            │
│        机理: 从多模 anchor 截断扩散, 2 步去噪 → 多模轨迹 → top-1   │
│        输出: 未来路点轨迹                                         │
└───────────────────────────────┬───────────────────────────────┘
                                 │  路点
                                 ▼
┌───────────────────────────────────────────────────────────────┐
│  控制器   差速跟踪器: 路点 → (v, ω) + 传感器避障   🟡(取决于底盘)  │
└───────────────────────────────────────────────────────────────┘
```

**双脑解决延迟**：VLM 慢（~5s/次），但 DiffusionDrive 2 步去噪、实时跑；下层始终用 VLM
**最近一次**的 task id 高频规划，VLM 异步更新、不阻塞。VLM 推理那几秒，下层保持安全动作
（慢速沿当前路径 / 原地转）。这就是团队要的「双脑」，DiffusionDrive 的速度让它成立。

## 2. DiffusionDrive 关键事实（hustvl, CVPR2025 Highlight, arXiv 2411.15139）

- **截断扩散**：不从纯噪声去噪，而是从一组**先验多模 anchor**出发；训练截断 50/1000 步，推理**仅 2 步**取 top-1。
- **级联扩散解码器**：与**条件场景上下文（BEV/感知特征）**做 cross-attention。← 这是注入 VLM 任务条件的接口。
- ResNet-34 backbone，NAVSIM 88.1 PDMS，**45 FPS@4090**，学人类示教、无后处理。
- 易于与车载传感器 + 现有感知模块集成；已开源 nuScenes / NAVSIM 权重与训练脚本。

## 3. 核心创新点：任务条件注入

- 原版只条件于「感知 BEV + ego」→ 沿车道开。
- 迁移版**多加一路 VLM 任务 id 条件**，注入级联解码器的 cross-attention →
  从「沿路开」变成「**按当前任务选轨迹模式**」。
- ✅ 任务条件用**离散任务 id**（5-10 类），理由：简单、抗幻觉、省数据，契合「VLM 只给区分度」。

## 4. 车端 → 机器人端：4 个迁移难点

1. **任务条件接口**（=创新点，见 §3）。✅ 离散 id。
2. **运行时 BEV 从哪来**（最大工程缺口）：
   - 车端有多相机+LiDAR 实时出 BEV；我们的 VGGT-Ω 出 BEV 是**离线、4.5GB**，Orin 上跑不了。
   - 解法：**先在 GS-Playground 仿真里训**（仿真免费给干净 BEV/深度）；真机 BEV 再单独训个轻量 LSS 编码器。🟡 真机方案待定。
3. **anchor 重聚类**：车的 anchor 来自驾驶轨迹（阿克曼/大尺度）；机器人室内是差速、慢、急转、能原地转 →
   用**我们采集的巡检轨迹** k-means 出机器人运动 anchor。
4. **数据不够 → 仿真造示教**：DiffusionDrive 学人类示教，真机示教少 →
   在 GS-Playground 里用脚本专家/遥操批量造示教训下层，再 sim2real。

## 5. 名词澄清（讨论中确认的两点）

- **轨迹示教来源**：DiffusionDrive 是模仿学习，需要大量〔观测 → 专家轨迹〕范例（=示教）。来源三选：
  | 来源 | 优点 | 缺点 |
  |---|---|---|
  | 真机 bag | 真实、无 sim2real 差 | 量少、不一定干净、无任务标签 |
  | **仿真脚本专家**（建议为主） | 量大、干净、能自动标任务 id | 有 sim2real 差 |
  | 遥操 | 人类意图、质量高 | 费人力 |
  → 💡 倾向：**仿真脚本专家为主 + 真机 bag 校准**。
- **下层动作表示**：DiffusionDrive 吐什么、机器人拿什么执行。
  - (a) 输出路点 + 加差速跟踪器转 (v,ω)  ← 💡 倾向（迁移改动最小）
  - (b) 直接输出 (v,ω) 序列（要改输出头）
  - 关键差异：车不能原地转，差速机器人能（「原地转圈」）；动作表示要能表达原地旋转。🟡 取决于底盘。

## 6. 训练策略：两阶段

1. **阶段一**：VLM 场景/任务识别先调到够准——**先 few-shot prompt、不训**，验证区分度够不够。
2. **阶段二**：识别够准后，再训「连起来」的部分——**下层不同场景的轨迹 policy**（DiffusionDrive）。
- 数据少：VLM 端若要训用 **LoRA** 足够；重点是把**场景间区分度**喂进去。
- 说话人4 担心 3B 幻觉 → 场景识别是窄分类任务，可**约束输出**（受限解码 / 限定任务标签集）降幻觉。

## 7. 部署与算力约束

- **端侧 = Jetson Orin**（不是 3090，弱很多）。大 VLM 上不了 Orin → 要么小模型(Qwen3-3B)量化本地、要么 API 联网。🟡 Orin 型号待确认。
- 训练端：6×A100-80G（不能全占）。
- VLM 选型：💡 先 Qwen3-3B/4B few-shot 验证；不够再考虑换更大 / 或退回传统特征提取（任务不复杂时可接受）。
- 延迟：VLM ~5s/次（6-8 图，开 CoT 更慢）；靠双脑掩盖。传统特征提取快很多，是 fallback。
